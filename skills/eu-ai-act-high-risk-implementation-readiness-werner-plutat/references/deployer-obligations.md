# Deployer Obligations (Art. 26)

## Purpose

Article 26 focuses on organizations that **use** high-risk AI systems in their operations. Many companies assume the provider carries the compliance burden. That is wrong.

Deployers have their own operational obligations, especially around proper use, oversight, data relevance, monitoring, record-keeping, and information duties.

The practical question is:

**Can the deployer show that the system is used as instructed, overseen by accountable humans, and monitored in live operation with proper records?**

---

## What deployers need in practice

A deployer readiness program usually covers:
- use in line with provider instructions for use
- assignment of human oversight responsibilities
- relevant and sufficiently representative input data where the deployer controls inputs
- monitoring of operation and anomalies
- retention of logs/records where required
- informing affected persons where applicable
- cooperation with provider obligations where issues arise
- fundamental rights impact assessment for public bodies in relevant cases

---

## Key deployer questions

### 1. Are we using the system within its intended purpose?
If the business uses the system outside the provider’s instructions or extends it in practice, deployer risk increases fast.

### 2. Who is the human overseer?
Not just “the business team.” Name actual roles, authority, escalation routes, and intervention rights.

### 3. Are inputs suitable?
If the deployer provides live inputs, poor quality or irrelevant data can create compliance and performance problems.

### 4. Do we monitor how the system behaves in real use?
There should be a way to capture errors, anomalies, complaints, override patterns, or drift indicators.

### 5. Are affected persons informed where required?
Depending on the use case, individuals may need to be informed that they are subject to the use of high-risk AI.

### 6. Is a FRIA needed?
For public authorities or public bodies using relevant high-risk systems, a **fundamental rights impact assessment** may be required.

---

## Evidence that should exist

Strong deployer evidence may include:
- provider instructions for use on file
- internal SOP for use of the system
- named oversight assignments
- operator training records
- input data rules/checks
- monitoring log or exception tracking
- incident/escalation workflow
- record retention logic
- notices to affected persons where relevant
- FRIA file where required

---

## Readiness scoring guide

### RED
- deployer assumes provider handles everything
- no internal operating procedure
- oversight not assigned
- business uses system beyond documented intended purpose
- no monitoring or records approach

### AMBER
- instructions are available, but not operationalized internally
- oversight exists informally
- some monitoring occurs, but not systematically
- information duties and FRIA questions not fully assessed

### GREEN
- use is aligned with provider instructions
- oversight roles and escalation are defined
- inputs, records, and monitoring are governed
- affected-person and FRIA issues are addressed where relevant
- incidents feed back internally and to the provider where needed

---

## Common pitfalls

### 1. “We are only the user”
That mindset creates blind spots. Deployers have real compliance duties.

### 2. Oversight without authority
If operators can see issues but cannot stop or escalate effectively, oversight is weak.

### 3. No adaptation to local operations
Provider instructions often need to be translated into local SOPs, training, and controls.

### 4. Informal monitoring only
“People would notice if something went wrong” is not a monitoring framework.

### 5. Ignoring employee/works council implications
In HR and workplace-related uses, employee representation issues can become operational blockers, especially in Germany.

---

## Practical next steps

### If RED
1. Confirm provider documentation and intended purpose
2. Write an internal use SOP
3. Assign human oversight roles
4. Define monitoring and escalation workflow
5. Assess information duties and FRIA trigger

### If AMBER
1. Tighten input data checks
2. Formalize operator training and record-keeping
3. Add incident and exception tracking
4. Align with provider updates and limitations
5. Validate works council/public body obligations if relevant

---

## Deployer operating model template

Document:
- system used
- provider
- intended purpose in local operations
- named users/operators
- named oversight function
- escalation triggers
- stop/override procedure
- input data controls
- logs/records retained
- notices/information duties
- FRIA status
- review cadence

---

## What authorities or auditors will care about

They will often focus on whether the deployer:
- stayed within intended purpose
- had meaningful human oversight
- kept adequate operational records
- responded to problems responsibly
- considered affected persons and fundamental rights where required

That is the practical core of Art. 26.
