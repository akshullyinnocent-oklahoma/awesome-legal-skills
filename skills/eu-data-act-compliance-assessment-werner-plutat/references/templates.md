## 2. User Data Access Request Response Template

Use when responding to user requests for data access under Article 4.

---

**Subject:** Your Data Access Request - [Product/Service Name]

Dear [User Name],

Thank you for your request dated [date] to access data generated by your use of [connected product/service name].

**Request Summary:**
- Product/Service: [name and ID]
- Data requested: [describe scope]
- Format requested: [JSON/CSV/other]

**Response:**

We are pleased to provide access to your data as follows:

**Access Method:**
- **Self-service portal:** [URL] (login with your existing credentials)
- **API endpoint:** [URL] (authentication: [method])
- **Data export download:** [link to download] (available for 30 days)

**Data Scope:**
The following data is included:
- [Data type 1]: [description, date range]
- [Data type 2]: [description, date range]
- [Additional types as applicable]

**Format:**
Data is provided in [JSON/CSV/XML] format. [Attach schema documentation / link to API docs].

**Frequency and Updates:**
- This is a [one-time export / real-time access].
- Data is updated [continuously / daily / as applicable].
- You may request updated exports at any time via [portal/API/email].

**Trade Secret Limitation (if applicable):**
[Note: Only include if withholding some data on trade secret grounds]

Certain data elements [describe which] have been [aggregated/redacted/withheld] to protect our trade secrets under Article 5(4) and 6 of the Data Act. Specifically:
- [Explain what is withheld and why]
- [Explain harm that would result from disclosure]
- [Describe safeguards considered and why insufficient]

We have provided all data that can be disclosed while protecting our legitimate commercial interests.

**GDPR Notice (if personal data included):**
[Include if data contains personal data]

This data includes your personal data. Under GDPR, you have the right to rectification, erasure, restriction, objection, and data portability. To exercise these rights, contact [DPO email/portal].

**Questions or Issues:**
If you have difficulty accessing the data or believe data is missing, please contact us at [support email/portal].

Best regards,
[Data Access Team]
[Company Name]

---

**Internal Checklist (not sent to user):**
- [ ] Verify user identity and authorization
- [ ] Extract data from systems
- [ ] Apply any necessary trade secret protections
- [ ] Generate export file or enable API access
- [ ] Test that data is complete and machine-readable
- [ ] Log request and response for audit trail
- [ ] Set calendar reminder for follow-up if access is time-limited

## 3. Third-Party Data Sharing Request Response Template

Use when responding to user requests to share data with a third party under Article 5.

---

**Subject:** Third-Party Data Sharing Request - [User] to [Data Recipient]

Dear [User Name],

Thank you for instructing us to share data from [product/service] with [data recipient name].

**Request Summary:**
- Data recipient: [name, contact, legal entity]
- Data scope: [describe what data user requested to share]
- Purpose: [describe the value-added service the recipient will provide]
- User authorization: [date of user instruction, confirmation method]

**Response:**

We have processed your request as follows:

**Option A - Direct Access:**
We have enabled [data recipient name] to access your data via secure API.
- API endpoint: [URL]
- Access credentials: [provided directly to data recipient]
- Access period: [ongoing / until [date]]

**Option B - Data Copy:**
We have provided a copy of your data to [data recipient] via [secure file transfer/email/portal].
- Data format: [JSON/CSV/XML]
- Date range: [from - to]
- File reference: [ID or link]

**Use Limitation:**
Under Article 5(1) of the Data Act, [data recipient] may use this data only for the purpose agreed with you: [state purpose]. They may not use it to develop a competing product or for other purposes without your further consent.

**Trade Secret Safeguards (if applicable):**
We have applied the following safeguards to protect our trade secrets while enabling your data sharing:
- [Describe: aggregation, redaction, confidentiality agreement, access restrictions]
- [Data recipient] has agreed to maintain confidentiality.

**GDPR Notice (if personal data included):**
This data includes your personal data. [Data recipient] is an independent data controller and must comply with GDPR. Their privacy notice is available at [link]. You have the right to withdraw consent or object to processing - contact [data recipient] directly.

**Questions:**
If you wish to modify or revoke this data sharing arrangement, contact us at [email].

Best regards,
[Data Access Team]
[Company Name]

- Data requested: [describe scope from request]
- Purpose: [emergency response / prevention / recovery]

## 5. Contract Review Checklist - B2B Data Sharing Agreements

Use this checklist when drafting or reviewing B2B contracts for data access or use.

---

### Unfair Terms Checklist (Article 13)

Review each term against the unfairness catalogue. Mark YES if the term is present, then assess whether it's unfair.

#### Unfair Per Se (Article 13(1)) - Automatically Void

| Term | Present? | Unfair? | Action Required |
|------|----------|---------|-----------------|
| Unilateral right to impose, alter, or terminate data access/use obligations | Y/N | If YES → VOID | Delete or make mutual |
| Unilateral right to interpret contract terms | Y/N | If YES → VOID | Replace with neutral dispute resolution |
| Exclusion/limitation of liability for intentional acts or gross negligence | Y/N | If YES → VOID | Exclude only liability cap, not liability itself |
| Exclusion/limitation of liability for breach of contractual obligations | Y/N | If YES → VOID | Allow reasonable liability cap, not exclusion |
| Unilateral termination without reasonable notice (except for cause) | Y/N | If YES → VOID | Add minimum notice period (e.g., 60 days) |

#### Presumptively Unfair (Article 13(2)) - Rebuttable Presumption

| Term | Present? | Unfair (presumed)? | Rebuttal Available? | Action Required |
|------|----------|-------------------|---------------------|-----------------|
| Inappropriately limits remedies for non-performance | Y/N | If YES → presumed unfair | Can rebut if proportionate | Revise remedies or document proportionality |
| Excludes or limits party's rights to contract data | Y/N | If YES → presumed unfair | Can rebut if IP protection justified | Revise to allow data retention or justify |
| Permits disproportionate or inadequately remunerated data use | Y/N | If YES → presumed unfair | Can rebut if fair value exchange | Add compensation or limit use scope |

#### Safe Harbor Assessment (Article 13(3))

| Safe Harbor | Applies? | Evidence |
|-------------|----------|----------|
| Terms were individually negotiated | Y/N | [List: email exchanges, redlines, meeting notes, legal counsel involvement] |
| Terms based on model terms developed/endorsed by representative organizations | Y/N | [Cite: industry association, multi-stakeholder body, public availability] |
| Terms aligned with Commission Model Contractual Terms (Article 41; Recommendation November 2025) | Y/N | [Identify which MCT set was used and document any deviations] |

**Overall Assessment:**
- [ ] Contract contains no unfair per se terms (or unfair terms have been removed)
- [ ] Presumptively unfair terms are either removed or rebuttal evidence documented
- [ ] Safe harbor applies (individual negotiation or model terms), OR contract has been revised to remove unfair terms
- [ ] Contract is compliant with Data Act Chapter IV

**Recommended Revisions:**
[List specific clauses to delete, revise, or add]

## 7. Cloud Switching Compliance Checklist

Use this checklist if you are a data processing service provider (cloud/edge).

---

### Switching Assistance (Article 23)

- [ ] **Customer can switch to another provider** without contractual prohibition
- [ ] **Customer can transition to on-premises** infrastructure
- [ ] **Self-service data export tool** available (portal or API)
- [ ] **Data export includes:**
  - [ ] All customer data
  - [ ] Applications and configurations
  - [ ] Metadata and logs (where customer-controlled)
  - [ ] Encryption keys (where customer-controlled)
- [ ] **Export format is:**
  - [ ] Structured (organized logically)
  - [ ] Commonly used (JSON, CSV, SQL, OVF/VMDK, container images, etc.)
  - [ ] Machine-readable (programmatically parseable)
- [ ] **Functional equivalence:** Data/apps can be used in target environment without loss of core functionality
- [ ] **Export is secure:** Encrypted in transit, authenticated access, logging

### Termination Notice (Article 24)

- [ ] **Minimum 2-month notice period** for customer termination
- [ ] **Minimum 2-month notice period** for provider termination or non-renewal
- [ ] **No minimum contract duration** that is longer than objectively necessary
- [ ] If minimum duration exists: **justification documented** (e.g., infrastructure investment, regulatory requirements)

### Fee Phase-Out (Article 25)

| Contract Type | Fee Status | Deadline | Compliant? |
|---------------|------------|----------|------------|
| **New contracts** (concluded after 12 Sept 2025) | Should have ZERO switching/exit fees | Immediately | Y/N |
| **Existing contracts** (concluded before 12 Sept 2025) | Must eliminate fees by Sept 2027 | 12 Sept 2027 | Y/N |

- [ ] **Switching fees eliminated** (or plan in place for Sept 2027)
- [ ] **Data export fees eliminated** (or plan in place for Sept 2027)
- [ ] **Early termination penalties eliminated** (except legitimate cost recovery)
- [ ] **Pro-rata charges** for services used until termination are permissible (not classified as exit fee)

### Documentation and Transparency

- [ ] **Switching guide** or migration playbook published (how to export data, what formats, target compatibility)
- [ ] **Customer communications** clearly explain switching rights, export procedures, notice periods, fees (if any legitimate costs)
- [ ] **Contract terms** reflect Data Act compliance (no prohibited fees, reasonable notice, export rights)

**Overall Compliance Status:**
- [ ] Fully compliant
- [ ] Partially compliant - action plan required (specify timeline and responsible owner)
- [ ] Non-compliant - immediate remediation required

## 8. Implementation Roadmap Template

Use this timeline to plan Data Act compliance rollout.

---

### Phase 1: Assessment (Months 1-2)

- [ ] Identify Data Act roles (manufacturer, data holder, cloud provider, etc.)
- [ ] Map products/services in scope
- [ ] Conduct gap analysis (use Gap Analysis Matrix above)
- [ ] Prioritize obligations by deadline and risk

**Deliverable:** Gap analysis report, priority list

### Phase 2: Design (Months 3-4)

- [ ] Design user data access mechanisms (APIs, portals)
- [ ] Design third-party sharing workflows
- [ ] Review and revise B2B contract templates
- [ ] Design B2G request intake and response process (if applicable)
- [ ] Design cloud export and switching mechanisms (if applicable)
- [ ] Integrate with GDPR, AI Act, CRA compliance programs

**Deliverable:** Technical specifications, process flows, contract templates

### Phase 3: Build (Months 5-7)

- [ ] Develop APIs, portals, export tools
- [ ] Implement security controls (authentication, encryption, logging)
- [ ] Update contracts and terms of service
- [ ] Train staff (legal, technical, support teams)
- [ ] Establish governance (data access review board, trade secret assessment team)

**Deliverable:** Functional systems, trained teams, updated contracts

### Phase 4: Test (Month 8)

- [ ] Internal testing of data access and export mechanisms
- [ ] User acceptance testing (pilot with friendly customers)
- [ ] Penetration testing and security review
- [ ] Contract review and legal validation
- [ ] Cross-regulation integration testing (GDPR, AI Act, CRA)

**Deliverable:** Test reports, validation sign-off

### Phase 5: Deploy (Month 9-10)

- [ ] Launch user data access and third-party sharing capabilities
- [ ] Publish transparency information (pre-sale disclosures, switching guides)
- [ ] Rollout updated contracts (new agreements, amendments for existing)
- [ ] Communicate to customers (email, website, support documentation)

**Deliverable:** Live systems, customer communications

### Phase 6: Monitor and Improve (Ongoing)

- [ ] Monitor data access request volume and response times
- [ ] Track user satisfaction and identify friction points
- [ ] Review trade secret refusals for consistency and justification
- [ ] Update processes based on regulatory guidance and enforcement actions
- [ ] Annual compliance audit

**Deliverable:** Monitoring dashboards, audit reports, continuous improvement log
